# Orphaned storage garbage collection

Fresh stores each model under a generated storage ID in:

```text
<database root>/models/<storageId>
```

Durable restore writes replacement snapshots under new storage IDs before it
commits the next manifest generation. A reset before the manifest commit, or an
interrupted best-effort cleanup after the commit, can therefore leave
unreferenced model directories behind. Those directories are harmless to
database correctness, but they consume LittleFS space.

Fresh removes this storage through startup and manual garbage collection.

## Startup collection

After `init()` loads and validates the durable manifest, Fresh performs
best-effort garbage collection before starting the sync task.

Startup collection:

1. reads both manifest slots;
2. selects the highest valid generation;
3. builds the complete set of referenced storage IDs;
4. scans the direct children of `<database root>/models`;
5. removes only unreferenced Fresh storage directories.

A cleanup failure does not prevent startup. It is exposed through
`FreshDiagnostics::garbageCollection`.

```cpp
FreshResult initialized = db.init("/fresh_app");
FreshDiagnostics diagnostics = db.diagnostics();

if (diagnostics.garbageCollection.degraded) {
    Serial.println(diagnostics.garbageCollection.message.c_str());
}
```

## Manual collection

Applications may run the same operation explicitly:

```cpp
FreshGarbageCollectionResult result;
FreshResult collected = db.collectGarbage(result);

Serial.printf(
    "scanned=%u removed=%u reclaimed=%u failed=%u\n",
    static_cast<unsigned>(result.scannedDirectories),
    static_cast<unsigned>(result.removedDirectories),
    static_cast<unsigned>(result.reclaimedBytes),
    static_cast<unsigned>(result.failedDirectories)
);
```

`collectGarbage()` first forces the current registry to a durable checkpoint.
It then serializes the scan with regular persistence and durable restore.

## Result fields

| Field | Meaning |
| --- | --- |
| `scannedDirectories` | Direct child directories examined under the models directory. |
| `removedDirectories` | Unreferenced Fresh storage directories removed completely. |
| `reclaimedBytes` | Reduction in LittleFS used bytes during candidate cleanup. |
| `failedDirectories` | Eligible directories that could not be removed. |

`FreshResult::affectedCount` is the number of directories removed.

## Safety rules

Garbage collection is deliberately conservative:

- no deletion occurs when neither manifest slot exists;
- no deletion occurs when an existing manifest slot is unavailable;
- no deletion occurs when no valid manifest generation can be loaded;
- a corrupt slot may be ignored only when another valid slot exists;
- only direct child directories with an exact 16-character lowercase
  hexadecimal storage ID are eligible;
- files are never treated as storage directories;
- non-Fresh directories and unknown application files are untouched;
- every storage ID referenced by the selected manifest is preserved.

The exact storage-ID rule matches IDs generated by Fresh. It prevents a damaged
manifest or an unrelated directory name from broadening the deletion scope.

## Power-loss behavior

Garbage collection is idempotent. A reset during deletion can leave a partially
removed unreferenced directory, which is retried on the next startup or manual
collection. The active manifest is never modified by collection, and referenced
model storage is never selected as a candidate.
